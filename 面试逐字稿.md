## 一、开场介绍（约 2 分钟，详细版）

"我做了一个叫 **DSVisualizer** 的数据结构可视化工具，是用 **C++17 和 Qt 5.12** 写的，开发过程是 **AI 辅助编程**完成的。

做这个工具的出发点，是我准备算法面试的时候，想找一个能把链表、栈、队列、各种平衡树、堆、哈希、图这些结构'动起来'的东西——很多现成工具要么太简陋、要么不开源、要么不能一步步看算法过程，所以就自己写了一个。

整个工具覆盖了 **16 种常用数据结构**：单/双链表、栈、队列、双端队列、阻塞队列、环形缓冲、二叉搜索树、AVL、红黑树、B 树、B+ 树、最小堆、哈希表，还有无向图和有向图。它的核心不是简单地把'最终结果'画出来，而是把每个操作——比如往红黑树里插一个节点、触发一次旋转——**拆成很多个'微步骤'，一步一步用动画演示出来**，这样能清楚看到旋转、分裂、借位这些过程到底发生了什么。

架构上我分了三层：
- 最底下是**纯 C++17 写的数据结构引擎**，完全不依赖 Qt，只负责算数据和产生'帧快照'；
- 中间是**渲染层**，用 Qt 的 Graphics View 框架自己画图元；
- 最上面是**交互层**，做下拉选择、播放控制、撤销重做这些。

三层之间靠一个'**帧快照（Frame）**'机制串起来：数据层每做一次操作，就吐出一串 Frame，每一帧记录了这一刻所有节点的位置和状态；渲染层拿这些 Frame 做差分动画。这个设计让我能把'算法正确性'和'画面怎么画'彻底解耦，算法可以单独测试。"

## 二、高频追问应答

### Q1：动画是怎么实现的？为什么不用视频或 GIF？
"如果用视频或 GIF，每一步都是死的，用户不能自己输入数据、不能暂停看某一帧、也不能回退。所以我用的是 Qt 的 **Graphics View 框架**——自己写 `QGraphicsObject` 的子类，重写 `paint()` 方法把节点画成圆角矩形、把边画成连线。

数据层每次操作会产生一串'帧快照'，每一帧记录这一刻所有节点的位置和状态。渲染层拿到前后两帧做'**差分**'：只把变化的节点移动过去、新增的建出来、删除的移除掉。移动过程用 `QPropertyAnimation` 做补间，位置、颜色、缩放都能平滑过渡，大概 340 毫秒一帧，目标 60 帧每秒。这样既能逐帧回放，又能平滑过渡，比视频灵活得多。"

### Q2（重点）：你刚提到树，红黑树/AVL 的旋转你是怎么保证不出错的？讲讲踩过的坑。
"这个恰恰是我这个项目里最有意思的一个 bug，也是花时间最多的一个。

**现象**：树一触发旋转就直接崩溃，报 SIGSEGV 段错误。

**根因**：最初我用 `unique_ptr` 管理树节点——每个节点用 unique_ptr 持有它的左右孩子，析构时不用手动管内存。但树要旋转，旋转就涉及把子树的所有权从一个指针 move 到另一个指针。问题出在旋转函数里：我先写 `x->right = std::move(y)`，把 y 的所有权移走，紧接着下面又去访问 `y->left`。但 **move 之后，原来的 y 这个 unique_ptr 已经变成空壳了，它内部指针是 null**，这时候再 `y->left` 就是解引用空指针，直接段错误。

**为什么难查**：这个 bug 只在 **MinGW 严格的所有权语义**下炸。我一开始用 Python 写了个对照测试，结果 Python 没有 move 语义，变量被'移走'之后还能访问，所以根本测不出来，还给了我一种'算法是对的'的错觉——这其实是个坑，纯 C++ 的 move 语义问题，脚本语言测不出来。

**修复**：在 move 之前，先用一个裸指针把地址存下来，比如 `Node* y_raw = y.get()`，move 之后所有对节点的访问都走这个裸指针 `y_raw`，而不是走已经被掏空的 unique_ptr。同时我把架构改成'**parent 裸指针导航 + 原地旋转**'——旋转只在很小的函数内部 move，move 完立刻放回原位，外面全程用 parent 指针找位置，这样 unique_ptr 的所有权转移窗口几乎为零，从根上消除了悬空解引用。

**验证**：为了确认修对了，我把整棵树的算法用 Python **忠实移植**了一份——严格模拟 unique_ptr 的 move 语义，包括'通过已 move 的空指针解引用就报 SIGSEGV'——然后跑了 **3000 多回合随机插入删除混合测试**，全部通过。这之后红黑树和 AVL 的旋转就稳了。"

### Q3：那你为什么用 unique_ptr + 裸 parent 指针，不用 shared_ptr？
"因为树里每个节点的所有权是唯一的——一个节点只有一个父亲拥有它。用 shared_ptr 的话引用计数永远是 1，白白增加开销，而且它也表达不出'谁才是 owner'这个语义。我的做法是：**孩子用 unique_ptr**（明确表达所有权、自动释放），**parent 用原始指针**（只用来往上导航，不参与释放）。这样职责很清楚。代价就是旋转时要小心 move 语义，但也正是刚才那个 bug 让我把这套纪律摸透了——move 前必存裸指针。"

### Q4：你说"帧快照差分"，跟每步直接重画整棵树比，好在哪？
"两个好处。一是**性能**：一棵树可能几十上百个节点，如果每步都把整棵树删了重建，会闪、也慢；差分只动变化的那几个节点。二是它能支撑'**逐帧回放**'——因为每一帧都是独立的快照，我可以把一串 Frame 存下来，用户想看第几步就看第几步，还能前进后退、撤销重做。这其实是整个工具能'教学'的基础。"

### Q5：16 种结构都是你自己从零实现的？怎么测的？
"对，核心结构都是自己写的，没用现成容器代替——因为目的就是展示算法本身。测试分两块：一部分是纯 C++ 的单元测试，每种结构测插入删除查找的正确性；另一块就是对树这种容易出错的，我用 Python 做'忠实移植'压力测试，随机混合几千轮操作，保证结构不损坏。可视化层面因为没有编译器我没法在本地跑，是依赖用户环境测的。"

### Q6：这个项目你最大的收获/难点是什么？
"最大的难点就是那个 **unique_ptr 旋转崩溃**——它逼我把 C++ 所有权语义、树算法、还有怎么在没有编译器的情况下做验证，这三件事串起来了。其次是怎么把'动画'和'算法正确性'解耦：我让数据层完全不碰 Qt，只吐快照，这样算法逻辑可以单独测试，不会因为渲染出问题而怀疑算法。这个'**引擎和渲染解耦**'的设计我觉得是项目里最值钱的决定。"

### Q7（收尾，诚实版）：这个项目有什么不足？
"有两点我如实说：一是**主题切换**、还有一些树（比如 AVL/红黑树）的 **BFS/DFS 遍历展示我还没做**，目前遍历顺序展示只在图和二叉搜索树上实现了；二是可视化部分我没法在本地编译验证，是依赖用户环境测的。但这些反而让我更清楚下一步往哪改。"

### Q8（你重点要求的）：这个项目是怎么用 AI 辅助开发的？用了什么 skill、大模型、arg、MCP？
"这个项目是 **AI 辅助编程**完成的，具体说是在一个带技能扩展机制的 AI 编程工作台里，通过**对话式编码（AI agent）**完成的——代码编写、静态审查、还有那些 Python 验证脚本，都是在工作台里由 AI 辅助产出的。

- **Skill（技能）**：工作台支持技能扩展机制，比如有 C++ 内存安全诊断、代码审查这类技能。我这个项目最难的 unique_ptr 旋转崩溃，正是用'严谨的所有权语义分析 + Python 忠实移植验证'这套方法攻克的，思路和内存安全诊断类技能一致。
- **大模型**：底层是**基于大语言模型的 AI 编程助手**，具体模型/厂商由所用平台决定。我在面试里就如实讲'用 AI 编程助手辅助开发'，不会去虚构某个具体版本。
- **arg（命令行参数）**：DSVisualizer 这个应用本身没有引入独立的命令行参数系统；如果是问开发过程，那就是由 **AI agent 对话驱动**，我下指令、它写代码和测试。
- **MCP（Model Context Protocol）**：**这个项目没有用到 MCP**。因为 DSVisualizer 是一个纯本地的 C++/Qt 桌面程序，不需要连接任何外部服务，工作台里的外部连接器（数据库、文档、代码托管等）当时都是断开状态，所以全程没接 MCP。"

## 附录：简历卡片（可复制）

**项目名称**：数据结构可视化工具 DSVisualizer ｜ AI 辅助开发 ｜ C++17 / Qt 5.12

**项目简介**：基于 C++/Qt 的数据结构可视化工具，将链表、栈、队列、平衡树（AVL/红黑树）、B/B+ 树、堆、哈希表、图等 16 种常用数据结构的插入/删除/查找以及图与二叉搜索树的深度优先遍历和广度优先遍历等核心操作，以分步动画形式呈现，辅助算法理解与学习。

**项目实现**：
- **分层架构设计**：设计引擎—渲染—交互三层架构，将数据结构与可视化解耦：数据层为纯 C++17、零 Qt 依赖；
- **可视化渲染**：基于 Qt Graphics View 框架自绘可视化，自定义 `QGraphicsObject` 图元重写 `paint()` 绘制节点与连线；
- **分步动画引擎**：设计 Frame 帧快照 + 差分动画机制，将旋转/分裂/借位等复杂操作拆为微步骤逐帧呈现；用 `QPropertyAnimation` 对节点位置/颜色/缩放做 340ms 补间，目标 60fps 平滑过渡；
- **数据结构与算法内核**：实现 16 种数据结构及 AVL、红黑树的插入/删除/旋转算法，采用 `unique_ptr` 持有节点 + 原始 `parent` 指针导航的旋转架构；
- **遍历与教学呈现**：在 Graph 与 BST 上实现 BFS/DFS 遍历顺序去重高亮展示，对标 VisuAlgo 的交互体验；
- **内存安全与工程验证**：定位并修复红黑树/AVL 旋转时 `unique_ptr` move 后悬空解引用导致的 SIGSEGV 根因崩溃，改用"move 前捕获裸指针 + parent 指针导航的原地旋转"彻底消除，并用 Python 忠实移植做 3000+ 回合随机混合压力测试验证。
