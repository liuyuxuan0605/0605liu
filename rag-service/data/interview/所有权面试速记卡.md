## 0. 核心设计（先背这个）

```cpp
struct Node {
    int val;
    std::unique_ptr<Node> left;    // ← 拥有型：独占左孩子，析构时自动释放
    std::unique_ptr<Node> right;   // ← 拥有型：独占右孩子
    Node* parent = nullptr;        // ← 非拥有：只向上导航，永不 delete
};
```

**一句话**：`unique_ptr` 当 owner（类型即文档「我拥有」），`raw` 当 observer（零成本、打断引用环）。这是 C++ Core Guidelines 推荐的树模式。

**关键不变量**：父节点用 `unique_ptr` 拥有子节点 ⇒ 父销毁子必销毁 ⇒ **子节点活着时，其父必然活着** ⇒ `parent` 在子存活期间永不悬空。

## 2. Q1：混用 unique_ptr + 裸 parent，会不会重复回收（double-free）？

**答案**：不会。double-free 需要「同一地址有两个所有者都去 `delete`」。
- 本设计里唯一所有者是 `unique_ptr`，整棵树析构只走 `unique_ptr` 一条递归路径，每个节点释放恰好一次。
- `parent` 是 non-owning observer，从不 `delete`，不参与释放 ⇒ 不存在「第二次释放」。

**真实发生的不是 double-free，是 use-after-move**（见第 5 节）：`std::move(y)` 后原 `unique_ptr` 内部指针变 null，再 `y->left = ...` 解引用 null ⇒ SIGSEGV。

**话术**：
> 不会。这里裸 `parent` 是 non-owning 观察者指针，只做向上导航，从不 `delete`；唯一所有者是 `unique_ptr`，整棵树析构只走 `unique_ptr` 一条路径，每个节点释放恰好一次。double-free 只会在「同一地址被两个 `unique_ptr` 持有」或「`unique_ptr` 外又手动 `delete`」时发生，这两种写法项目里都规避了。

---

## 3. Q2：`parent` 全程没释放，会不会内存泄露？

**答案**：不会。
- `parent` 只是 `Node` 里的一个字段（8 字节地址），**不是独立堆分配**。Node 被 owning `unique_ptr` 释放时，它跟着结构体一起没，不存在「只丢指针却没 free」。
- 泄露的真正来源是「拥有型裸指针 `new` 不 `delete`」或「`shared_ptr` 循环引用」，`parent` 两者都不是。
- **反直觉点**：用 raw `parent` 反而比 `shared_ptr` parent 更防泄露——若父子都用 `shared_ptr` 会成引用环、计数降不到 0；raw 打断了环。

**话术**：
> 不会。`parent` 只是 `Node` 里的一个字段，不是独立堆分配；`Node` 由 owning 的 `unique_ptr` 释放时它跟着一起没了。泄露只来自「拥有型裸指针忘了 delete」或「`shared_ptr` 循环引用」，`parent` 两个都不是。甚至反过来：用 raw `parent` 比 `shared_ptr` parent 更防泄露，因为 `shared_ptr` 父子互指会成引用环、计数降不到 0，而 raw 不打断计数、析构时整棵树干净递归释放。

---

## 4. Q3：那干脆全用裸指针，不用智能指针，不是更省事？

**答案**：更糟。裸指针把释放责任摊到**每一个会移除节点的地方**——旋转、删除、分裂、合并，漏删=泄露、删了再用=use-after-free（比 use-after-move 更隐蔽）、重复删=崩。
- `unique_ptr` 的坑是**局部的**（只在旋转函数里「move 前没存裸指针」，学会一次不再犯）。
- 裸指针的坑是**弥漫的**（几十处删除点，任何一处手滑就出事）。
- `unique_ptr` 把「销毁整棵树」这种最该不出错的路径，交给编译器免费保证。

**客观补充**：确有老练工程师偏好全裸指针树（觉得 unique_ptr 旋转别扭），但需铁律 ownership 约定 + 常配对象池。对教学工具没必要。另可选 **index-based** 树（节点存 `vector`、用下标互指），彻底无指针问题，但属架构级改动。

**话术**：
> 不省事，反而更危险。裸指针把所有释放责任摊到每个会移除节点的地方——旋转、删除、分裂、合并，漏删就泄露、删了再用是比 use-after-move 更隐蔽的 use-after-free、重复删直接崩。而 `unique_ptr` 把「销毁整棵树」交给编译器免费保证。我们踩的坑是局部的、只在旋转里「move 前没存裸指针」，学会一次就不会再犯；裸指针的坑是弥漫的。所以最优解是各司其职：`unique_ptr` 当 owner（类型即文档）、raw 当 non-owning 观察者（还顺手打断环）。

---

## 5. Q4：为什么不用 `shared_ptr` / `weak_ptr` 来省心？

**答案**：
- `shared_ptr` 孩子 + `shared_ptr` parent ⇒ **引用环 ⇒ 泄露**（经典陷阱）。
- `shared_ptr` 孩子 + `weak_ptr` parent ⇒ 能打破环、不泄露，**但划不来**：
  1. 树所有权本质是**单一**的（每节点恰一父拥有），`shared_ptr` 表达「共享」是范畴错误，多付引用计数开销且误导意图。
  2. **关键不变量**：子活 ⇒ 父必活，故 `parent` 在子存活期间永不悬空 ⇒ `weak_ptr` 那层「对象可能已死、先 `lock` 检查」的保护**完全冗余**。
  3. `weak_ptr` 只能从 `shared_ptr` 构造 ⇒ 想用它就得把**整棵树都改成 `shared_ptr` 管理**，反而引入复杂度。
  4. `unique_ptr` 的 move 是零开销指针交换；`shared`/`weak` 带控制块 + 原子计数。

**话术**：
> `shared_ptr` 当 parent 直接不行——父子互指成引用环，计数降不到 0，整树泄露。`weak_ptr` 当 parent 虽能打破环、也不泄露，但划不来：第一，树所有权本质单一，用 `shared_ptr` 表达「共享」是范畴错误还多付计数；第二，因为父用 `unique_ptr` 拥有子、父销毁子必销毁，所以「子活 ⇒ 父必活」，`parent` 永不悬空，`weak_ptr` 那层保护冗余；第三，`weak_ptr` 只能从 `shared_ptr` 造，想用就得全树改 `shared_ptr`，反而引入复杂度；第四，`unique_ptr` move 是零开销指针交换，`shared`/`weak` 带控制块和原子计数。所以最优仍是 `unique_ptr` 独占孩子 + raw 非拥有 parent。

## 6. 实战纪律：move 前先存裸指针（防 use-after-move）

```cpp
// ❌ 危险：move 后原 unique_ptr 内部指针变 null，再访问就 SIGSEGV
auto y = std::move(x->right);
x->right->left = ...;          // ❌ 解引用已为 null 的 x->right

// ✅ 安全：move 前先抓裸指针，之后走裸指针
Node* y_raw = x->right.get();          // 先存地址
x->right = std::move(y_raw->left);
if (x->right) x->right->parent = x;
y_raw->left = ...;                     // 走 y_raw，不碰已 move 的 unique_ptr
// 旋转后同步维护 parent（否则向上回溯平衡会走错）
if (y_raw->left)  y_raw->left->parent  = y_raw;
if (y_raw->right) y_raw->right->parent = y_raw;
y_raw->parent = x->parent;
```

**纪律口诀**：「move 前必 `.get()` 存裸指针；move 后只走裸指针；旋转后必对齐 parent。」

## 7. 30 秒速记口诀

> 拥有用 unique，观察用 raw；
> 释放只走一条路，double-free 无处来；
> parent 是字段不是堆，泄露轮不到它；
> 全裸坑弥漫，shared 成环泄；
> weak 虽安全，冗余又倒逼；
> move 前存裸指针，旋转对齐 parent。
