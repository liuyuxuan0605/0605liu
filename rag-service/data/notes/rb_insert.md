# 红黑树插入：为什么会"翻转"（旋转）

插入一个新节点后，你可能在可视化里看到树的结构突然"翻转"了一下——这就是**旋转（rotation）**。它只在特定条件下发生，不是每次插入都有。

## 插入的基线规则
新节点一律先染成**红色**插入。原因：红色不增加"黑高"（从根到叶子的黑色节点数），所以单独插入红色**不会破坏黑高相等**这条最核心的约束。只有后续出现冲突才需要修复。

## 什么情况下会触发修复
插入红色节点后，如果它的**父节点也是红色**，就违反了"红红不能相邻"规则，出现**红红冲突（red-red violation）**。这时必须修复。修复走两条路，取决于"叔叔节点（父节点的兄弟）"的颜色：

### 情况一：叔叔是红色 → 只重新着色，不旋转
把父节点和叔叔都染黑、祖父（爷爷）染红。红红冲突被消解，但冲突可能上移到祖父那一辈（祖父变红后若它父亲也是红，继续修）。
**这一步没有"翻转"，只是换颜色。**

### 情况二：叔叔是黑色（或 NIL）→ 必须旋转
此时单靠重新着色解决不了：祖父的另一个子树黑高已经正确，无法通过改色把局部平衡回来，只能**靠旋转改变树的结构**来恢复平衡。这就是你看到的"翻转"。根据新节点相对父、祖父的位置，分四种情形：

- **外侧插入（LL / RR）**：新节点在"祖父→父→自己"的同侧（都在左，或都在右）→ 对祖父做一次**单旋**（LL 右旋、RR 左旋）。
- **内侧插入（LR / RL）**：新节点在中间侧（左-右 或 右-左）→ 先对父节点做一次反向单旋，再对祖父做一次正向单旋，即**双旋**。

> 旋转前后**中序遍历必须保持不变**——这是验证旋转正确性的硬指标。旋转既不改变黑高，也正好把红红冲突解开。

## 回到你的问题：为什么"加了 66 会翻转"
当你插入 66 时，如果它落的位置形成了一个**红红冲突，且 66 的叔叔节点是黑色**（或不存在），算法就**只能旋转**才能恢复平衡——重新着色在这里无能为力。于是你看到树"翻转"了一下：一次单旋或双旋把局部结构重构，红红冲突被解开，黑高重新相等。

换句话说：**"翻转"= 叔叔为黑时的旋转修复**；如果叔叔是红色，那一insert只会重新着色、不会翻转。下次看到翻转，先确认被插入节点的叔叔是不是黑色就明白了。

## 面试一句话总结
"红黑树插入先染红保黑高；若父红（红红冲突）且叔叔红 → 只改色，叔叔黑 → 必须旋转。所以'翻转'就是叔叔为黑时的旋转修复，用来在不破坏黑高的前提下解开红红冲突。"
